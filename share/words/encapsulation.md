## 3줄 요약
- 코드를 수정할 때 가장 두려운 것은 나도 모르게 다른 부분에 영향을 미치는 것
- 영향을 주면 안되는 부분을 숨기자
- 변경될 수 있는 부분을 숨기자

## 자세히

### 왜 이 개념이 존재하는 것일까? (무엇을 해결하기 위한 걸까?)
- 한 번 만들고 끝나는 개발이 아니라면, 계속 변경되어야만 함(계속 변경되는 요구사항들)
    - 무엇 때문에 변경이 어려운걸까?
        - 내가 코드를 수정해도, 기존에 제대로 돌아가던 부분이 영향이 없어야 함
    - 어떻게 하면 다른 부분에 영향을 미치지 않도록 할 수 있을까?
        - 제한을 둔다..!! (컴터에서 되게 포괄적으로 적용되는 개념)

> '보이지 않으면 (존재하는지 조차 모르면) 건들수도 없으니.. 영향을 미칠 수 없다..!'
> 영향을 주면 안되는 부분을 숨기자..!!

### 어떤 경우에 이 개념이 적용되었을까?
- 클래스
    - 접근 제한자 
        - private 
    - 메서드 
        - 내가 요청하는 방식을 제한 (파라미터, 리턴값이 결정되었기에)
            - ex. 메서드 내부에서 지역변수를 가지고 무엇을 하든 외부에선 모름
                - 지역변수만 가지고 있다면 외부에서 코드 수정이 있어도 메서드 내부는 아무 영향이 없겠지?
- 인터페이스
    - 외부에서 사용하는 방식을 제한
        - 구현에 관한 세부 정보를 숨김
- 디자인 패턴
    - 각자 숨기고 싶은 부분들이 있음
        - ex. Strategy Pattern -> 어떤 구현체를 선택할지를 숨김
        - ex. Decorator Pattern -> 어떤 순서로 적용할지를 숨김


encapsulation?! 
> hide one part of the system from another part of the system. - Emergent Design (https://play.google.com/store/books/details?id=Q_3i19icWKMC&pcampaignid=books_web_aboutlink)
> in so doing we prevent one possible place where we might do unforeseen harm

일단 막 적어간다...

어떤 부분에서 중요하다고 생각하는 건지

내가 변경한 부분이 기존에 잘 돌아가던 부분에 영향을 줄까봐??



제한을 두는 것..!
- 위험할 수도 있는 부분을 신경 쓰지 않을 수 있도록

- 숨기면 어떤 것이 다를까요?
    - 영향을 줄 방법이 없음 (존재하는지 조차도 모르기에!)
        - 예를 들어 클래스 내부의 필드를 private 로 하면 -> 내부의 어떤 필드가 있는지 밖에서는 알 수 없음

- encapsulation 을 잘 했을경우?!
    - 변경으로 부터 자유로움
        - 다른 부분에 영향을 주지 않기에, 내가 구현할 부분만 신경 쓰면 됨

- 어떤 종류의 encapsulation 이 존재할까요?
    - 클래스라는 덩어리로 묶은 것
        - 접근 제한자
        - 메서드
    - interface 를 통한 추상화
        - 어떻게 요청을 보낼 건지
            - 상세(구현체)한 부분을 숨김
    - 디자인 패턴
        - 각자가 숨기고 싶은 부분들이 있음
